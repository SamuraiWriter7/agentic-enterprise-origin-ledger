# Enterprise Value Return Ledger

Version: 0.5.0

## 1. Purpose

The Enterprise Value Return Ledger binds verified enterprise value to an
approved Human Contribution Attribution record and records how value is
returned to contributors and shared organizational structures.

The ledger covers:

- verification of enterprise value;
- exclusion of speculative or double-counted value;
- creation of a policy-governed return pool;
- eligibility decisions;
- contributor allocations;
- collective and unattributed allocations;
- monetary and non-monetary return;
- holdbacks and appeal reserves;
- settlement status;
- privacy, employment, tax, and governance boundaries;
- audit integrity and lifecycle retention.

Version 0.5 completes the initial protocol arc:

```text
Enterprise Knowledge Origin
    ↓
Knowledge-to-Agent Skill Binding
    ↓
Cross-Department Execution Trace
    ↓
Human Contribution Attribution
    ↓
Enterprise Value Return Ledger
```

## 2. Core principle

> Verified value may return only through verified lineage.

A claimed business benefit must not become a contributor payment merely
because an AI system was involved.

The ledger requires:

1. verified enterprise value;
2. an approved and allocation-ready v0.4 attribution record;
3. an approved return policy;
4. explicit eligibility decisions;
5. balanced allocation;
6. review and appeal safeguards;
7. a traceable settlement route.

## 3. Value is not the same as return

Version 0.5 separates three quantities:

```text
Verified enterprise value
        ≠
Value-return pool
        ≠
Immediate settlement
```

For example:

```text
Verified enterprise value:  JPY 1,250,000
Approved return rate:        20%
Value-return pool:           JPY   250,000
Immediate value:             JPY   238,500
Held value:                  JPY    11,500
```

The organization may retain most of the verified benefit while returning a
policy-defined share to contributors and collective knowledge structures.

## 4. Normative requirements

A conforming Enterprise Value Return Ledger MUST:

1. reference an approved v0.1 origin record;
2. reference the applicable v0.2 skill binding;
3. reference one or more v0.3 execution traces;
4. reference an approved, allocation-ready v0.4 attribution record;
5. preserve the integrity digest of the attribution record;
6. declare the valuation period, scope, boundary, and currency;
7. distinguish included and excluded value components;
8. include only verified components in gross verified value;
9. record the valuation method and supporting evidence;
10. perform a double-counting check;
11. identify the approved return policy;
12. calculate the gross return pool under that policy;
13. record contributor eligibility decisions;
14. preserve v0.4 attribution weights or record an approved adjustment;
15. assign the unattributed share to a declared collective destination;
16. distinguish allocated, immediate, held, and settled value;
17. record settlement routes and status;
18. record non-monetary returns separately from monetary allocation;
19. provide notice and appeal safeguards;
20. preserve privacy and identity controls;
21. state employment, payroll, tax, and procurement boundaries;
22. prohibit settlement while a material appeal is unresolved;
23. balance all allocations against the return pool;
24. retain settlement evidence and superseding records.

## 5. Value verification

Every monetary component has:

- an effect type;
- a description;
- a monetary value;
- a verification status;
- an inclusion status;
- a measurement method;
- supporting evidence;
- an exclusion reason when it is not included.

An included component must have `verification_status: verified`.

Examples of included effects may include:

- avoided downtime;
- verified rework reduction;
- measured energy savings;
- realized revenue gain;
- confirmed quality improvement.

Examples of excluded effects may include:

- speculative reputation protection;
- unsupported future revenue;
- duplicate savings claims;
- generalized productivity assumptions;
- benefits outside the declared period.

Excluded effects remain visible in the ledger but do not enter the return pool.

## 6. Return policy

The return policy determines how verified value becomes a return pool.

Supported bases include:

- percentage of verified value;
- fixed pool;
- hybrid policy.

A percentage-based policy follows:

```text
Gross return pool =
    verified enterprise value × approved return rate
```

The policy may also define:

- minimum and maximum pools;
- attribution basis;
- holdback rate;
- allocation precision;
- the destination of unattributed contribution;
- governing policy references.

## 7. Eligibility

Contribution attribution does not automatically create settlement eligibility.

Eligibility may depend on:

- confirmed v0.4 contribution status;
- absence of an unresolved dispute;
- consent and identity restrictions;
- employment or contractual status;
- payroll, procurement, or fund rules;
- lawful settlement routes.

An ineligible contributor's weight must not silently disappear or be reassigned
to a visible executive. The ledger must record the approved treatment of that
share.

## 8. Recipient allocation

Each human or human-team allocation records:

- the v0.4 contribution ID;
- the protected recipient reference;
- the allocation weight;
- allocated monetary value;
- immediate monetary value;
- held monetary value;
- hold reason;
- settlement route;
- settlement status;
- scheduled time and settlement reference.

For a confirmed relative weight:

```text
Allocated value =
    gross return pool × allocation weight
```

And where a holdback applies:

```text
Held value =
    allocated value × holdback rate

Immediate value =
    allocated value − held value
```

## 9. Collective allocation

Not all contribution can be assigned fairly to individuals.

Collective allocations may support:

- shared enterprise knowledge funds;
- undocumented collective practice;
- common infrastructure;
- verification reserves;
- appeal reserves;
- future contributor discovery.

The unattributed share is not an error. It is an explicit statement that the
available evidence does not justify personal assignment.

## 10. Non-monetary return

Version 0.5 supports non-monetary value return such as:

- permanent origin recognition;
- protected time for knowledge capture;
- training credit;
- governance participation;
- knowledge-stewardship roles;
- access to tools or resources.

These returns must not be silently converted into monetary equivalents inside
the ledger.

Non-monetary recognition is especially important when the contributor values
continued authorship, time, or governance authority more than a one-time
payment.

## 11. Settlement states

Recipient settlement status may include:

- `pending_review`
- `authorized`
- `scheduled`
- `partially_settled`
- `settled`
- `held`
- `cancelled`

Ledger lifecycle status may include:

- `draft`
- `pending_review`
- `authorized`
- `partially_settled`
- `settled`
- `suspended`
- `voided`
- `archived`

Authorization is not the same as settlement.

A ledger may be balanced and authorized while settlement remains blocked by:

- an open appeal window;
- payroll verification;
- tax review;
- identity-protection controls;
- fund-eligibility checks;
- integrity verification.

## 12. Appeal and holdback

A holdback protects contributors and the organization from premature finality.

It may be released only after declared conditions are satisfied, such as:

- the appeal period closes;
- no material dispute remains open;
- the recipient route is valid;
- the ledger integrity digest is confirmed;
- no double counting or valuation error is found.

Holdback must not be used as an indefinite mechanism to avoid settlement.

## 13. Employment and legal boundary

The ledger supplies a traceable value-return basis. It does not override:

- employment law;
- payroll rules;
- taxation;
- procurement rules;
- collective agreements;
- intellectual-property law;
- fiduciary duties;
- accounting standards.

A value-return allocation must not be used to reduce ordinary salary, benefits,
or negotiated compensation.

## 14. Semantic balance requirements

A conforming implementation should verify that:

```text
included value components
    = gross verified value

verified value × policy rate
    = gross return pool

recipient allocation weights
    + collective allocation weights
    = 1.0

recipient allocations
    + collective allocations
    = gross return pool

immediate value
    + held value
    = allocated value

settled value
    + unsettled value
    = authorized allocation
```

The included validation script checks the principal arithmetic and cross-field
conditions.

## 15. Prohibited shortcuts

Version 0.5 prohibits:

- paying from estimated or disputed value;
- rewarding only the final visible approver;
- erasing the original contributor after payment;
- treating AI support as a fictitious human recipient;
- reallocating unattributed value without policy approval;
- settling while a material appeal is open;
- using value return as a substitute for ordinary compensation;
- treating the ledger as a complete employee-value score;
- inventing monetary equivalents for non-monetary recognition;
- counting the same business effect twice.

## 16. Completion of the initial arc

Version 0.5 completes the first repository arc:

```text
v0.1  Where did the enterprise knowledge originate?
v0.2  How was the knowledge transformed into an agent skill?
v0.3  What happened when the skill operated across departments?
v0.4  Which human contributions causally supported the outcome?
v0.5  How is verified value returned under an auditable policy?
```

The result is not merely a payment record.

It is a persistent causal chain from human knowledge to enterprise value and
back to human and collective contribution.
